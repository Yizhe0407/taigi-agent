# 任務交辦 Prompt 範本

> 用途：主對話派 subagent（Agent tool）時複製填空。每份範本含三要素：目標與動機、驗收條件、回報格式。
> 模型選擇見 `model-dispatch.md` 的調度表。`{...}` 是填空處。

## 通用規則（每個 prompt 結尾都附上）

```
回報規則：
- 只回結論 + 檔案:行號，不要貼大段檔案內容。
- 超過 30 行的產物先寫到檔案，回傳路徑。
- 做不到就明說「做不到，因為 {原因}」，不要編造結果。
- 回報上限 {N} 行。
```

## 1. 搜尋（Explore agent，model: haiku；範圍模糊或跨多目錄時 sonnet）

```
在 {目錄/整個 repo} 找 {目標：函式/設定/呼叫點/慣例}。
動機：{為什麼要找，讓 agent 能判斷相關性}。
搜尋廣度：{medium / very thorough}。

驗收條件：
- 每個結果附 檔案:行號。
- 明確說「找完了」或「還有 {範圍} 沒掃」。
- 找不到也要回報，並列出你試過的關鍵字。

回報格式：bullet list，每條一行，上限 20 行。
```

## 2. 實作（general-purpose，model: sonnet）

```
目標：{一句話}。動機：{為什麼做這件事}。
先讀：CLAUDE.md、{相關檔案清單}。照 CLAUDE.md「新增工具流程」的分層放檔案。

限制：
- {不可動的檔案 / 不可改的介面}
- Tool handler 必須回傳 str（如適用）。
- 不新增依賴；真的需要先停下回報。

驗收條件：
- cd backend && uv run pytest 綠、uv run ruff check . 乾淨（frontend 則 pnpm typecheck）。
- {功能驗收：具體輸入 → 預期輸出}
- 新邏輯有至少一個測試。

回報格式：
1. 改了哪些檔（檔案:行號）
2. 做了什麼、為什麼這樣寫、可能的坑（各一句）
3. 測試輸出最後 5 行
```

## 3. 重構（general-purpose，model: sonnet；跨模組邊界的重構先派 Plan agent）

```
目標：把 {現狀} 重構為 {目標狀態}。動機：{痛點}。
硬性限制：對外行為不可變 — 重構前先跑 uv run pytest 記下基準，重構後必須同樣全綠。
範圍：只動 {檔案清單}；發現需要動範圍外的檔案 → 停下回報，不要擅自擴大。

驗收條件：
- pytest 前後皆綠，無跳過（skip）新增。
- ruff 乾淨。
- diff 不含行為變更（無新分支邏輯、無改動測試預期值）。

回報格式：
1. 動了哪些檔（檔案:行號）
2. 結構前後對照（各 3 行以內）
3. 沒動但建議之後動的（如有）
```

## 4. 研究（general-purpose 或 claude-code-guide，model: sonnet；需要 WebSearch）

```
問題：{具體問題}。動機：{這個答案會決定什麼}。
查證優先序：官方文件 > 官方 repo/changelog > 其他。禁止憑訓練記憶回答版本號、參數名、價格。

驗收條件：
- 每個事實附來源 URL。
- 查不到的項目標「NOT FOUND」，不要推測。
- 有版本差異時註明適用版本與查證日期。

回報格式：問題逐條回答，每條 = 答案一句 + URL。上限 30 行。
```

## 5. 審查（general-purpose，model: sonnet；高風險 diff 用 opus）

```
審查對象：{diff / 檔案清單 / PR}。動機：{要防什麼——正確性 bug？過度設計？}。
你是 fresh context：不要假設寫的人是對的。先自己讀 {CLAUDE.md 相關 gotchas}，再對照。

必查清單：
- TDX direction 是否用 0/1（不是 1/2）。
- messages 截斷是否以 tool-call 輪次為單位。
- tool handler 是否回傳 str。
- 測試是否真的斷言行為（不是斷言 mock 被呼叫就完事）。
- {本次特別要查的點}

驗收條件：每個 finding = 位置(檔案:行號) + 問題一句 + 修法一句。沒問題就明說「查過 {清單}，無 finding」。

回報格式：finding list，嚴重的排前面。上限 25 行。
```

## 6. 驗證 / read-back（general-purpose，model: haiku）

```
你是 fresh context 驗證者，之前的對話你都沒看過（這是刻意的）。
讀 {檔案路徑}，然後回答：
1. 這份文件/程式碼要你做什麼？（用你自己的話，3 句內）
2. 有沒有指令互相矛盾、路徑不存在、或看不懂要怎麼執行的地方？逐條列。
3. {針對性問題，例：照這份文件，新增一個 provider 要動哪些檔案？}

驗收條件：第 2 題答「無」之前，必須先實際確認每個提到的路徑存在（ls / Read）。
回報格式：三題逐題答，上限 20 行。
```
